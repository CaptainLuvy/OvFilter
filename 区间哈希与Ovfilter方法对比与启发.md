## Unikmer + 链式扩展 + SD 的形式化定义与公式（可写进论文）

以下可直接或略作改写后放入第三章，作为「概念」与「公式」包装，增强方法部分的规范性。

### 概念名称建议

- **中文**：**基于一致 unikmer 链的 overlap 验证**，或 **UC-SD 准则**（Unikmer Chain with SD constraint）。
- **英文**：Unikmer-Chain SD Verification，或 UC-SD criterion。

### 符号约定

- \(Q\)：query read，\(T\)：target read；\(L_Q, L_T\) 为其长度。
- 重叠区域：\(Q\) 上 \([q_s, q_e)\)，\(T\) 上 \([t_s, t_e)\)；链方向 \(\sigma \in \{+, -\}\)（正向 / 反向）。
- \(U\)：参考基因组上的 unikmer 集合（出现次数为 1 的 k-mer 集合）。
- \(H_Q, H_T\)：\(Q\)、\(T\) 在各自重叠区内出现的 unikmer 的**有序**列表（按在 read 上的位置排序），每个元素为 (hash, position)。

### 定义 1（共享 unikmer 与重叠区内的出现）

称 unikmer \(u\) 在重叠区内**共享**，若 \(u\) 同时出现在 \(Q\) 的 \([q_s, q_e)\) 与 \(T\) 的 \([t_s, t_e)\) 中。记在 \(Q\) 上的位置为 \(p_Q(u)\)，在 \(T\) 上的位置为 \(p_T(u)\)。

### 定义 2（位置差与链方向）

对共享的 unikmer 对，定义**位置差**（与链方向一致）：

\[
\Delta(u) = p_Q(u) - p_T(u) \quad (\sigma = +); \qquad \Delta(u) = p_Q(u) + p_T(u) \quad (\sigma = -).
\]

（反向时用 \(p_Q + p_T\) 保证正确重叠下 \(\Delta\) 近似为常数。）

### 定义 3（共线延伸约束）

设链上相邻两对 unikmer 在 \(Q\) 上的位置为 \(p_Q^{(i)}, p_Q^{(i+1)}\)，在 \(T\) 上为 \(p_T^{(i)}, p_T^{(i+1)}\)。满足**共线延伸约束**（参数 \(\tau\)，即 tolerance）是指：

\[
\bigl| \, (p_Q^{(i+1)} - p_Q^{(i)}) - \sigma\cdot (p_T^{(i+1)} - p_T^{(i)}) \, \bigr| \le \tau.
\]

其中 \(\sigma = +\) 时 \(\sigma\cdot \delta = \delta\)，\(\sigma = -\) 时 \(\sigma\cdot \delta = -\delta\)。即两 read 上「步长」之差在 \(\tau\) 以内。

### 定义 4（一致 unikmer 链）

**一致 unikmer 链**（或**可扩展 unikmer 链**）\(C\) 是满足以下条件的共享 unikmer 对的极大集合：(1) 在 \(Q\)、\(T\) 上均按位置有序；(2) 链中任意相邻两对均满足共线延伸约束（定义 3）。链的**长度** \(|C|\) 为链中 unikmer 对的个数。

### 定义 5（链的位置差序列与标准差）

对一致 unikmer 链 \(C = \bigl((u_1, p_Q^1, p_T^1), \ldots, (u_n, p_Q^n, p_T^n)\bigr)\)，**位置差序列**为 \(\Delta_1, \ldots, \Delta_n\)（由定义 2）。其**链标准差**为：

\[
\mathrm{SD}(C) = \sqrt{ \frac{1}{n} \sum_{i=1}^{n} \bigl( \Delta_i - \bar\Delta \bigr)^2 }, \quad \bar\Delta = \frac{1}{n}\sum_{i=1}^{n} \Delta_i.
\]

### 判定准则（UC-SD 准则）

**给定参数**：最小链长 \(n_{\min}\)（min_shared）、共线容忍度 \(\tau\)（tolerance）、最大链标准差 \(\sigma_{\max}\)（max_sd）。

**重叠通过 unikmer 验证**当且仅当：存在至少一条一致 unikmer 链 \(C\)，使得

\[
|C| \ge n_{\min} \quad \text{且} \quad \mathrm{SD}(C) \le \sigma_{\max}.
\]

可写为一句判定句：「若存在长度不小于 \(n_{\min}\) 且链标准差不超过 \(\sigma_{\max}\) 的一致 unikmer 链，则接受该 overlap；否则拒绝。」

### 小结表（可作表 3-X）

| 符号 / 概念 | 含义 |
|-------------|------|
| \(U\) | 参考基因组 unikmer 集合 |
| \(H_Q, H_T\) | Q、T 在重叠区内的 unikmer 有序列表 |
| \(\Delta(u)\) | 位置差（与链方向 \(\sigma\) 一致） |
| 共线延伸约束 | 相邻 unikmer 在 Q、T 上步长之差 \(\le \tau\) |
| 一致 unikmer 链 \(C\) | 满足共线约束的极大共享 unikmer 对集合 |
| \(\mathrm{SD}(C)\) | 链 \(C\) 上位置差的标准差 |
| UC-SD 准则 | \(|C| \ge n_{\min}\) 且 \(\mathrm{SD}(C) \le \sigma_{\max}\) 则接受 |

这样可以把「unikmer + 链式扩展 + SD」包装成：**定义 1～5 + 一个判定准则 + 一个符号表**，便于在论文中引用（如「由定义 4，一致 unikmer 链满足共线延伸约束；由 UC-SD 准则，若存在满足长度与 SD 约束的链则接受该 overlap」）。

## 一、两种方法的核心区别

### 1. 你当前的方法：Ovfilter（基于独特 k-mer 重叠校验）

| 维度 | 内容 |
|------|------|
| **介入阶段** | **后处理**：对已有 PAF（如 minimap2 ava 输出）做过滤 |
| **输入** | PAF 文件 + unikmer 文件 + reads（FASTA/FASTQ） |
| **核心思想** | 用**仅出现一次的 k-mer（unikmer）** 做 overlap 校验。对每条 PAF，在重叠区内找两 read 的**共享 unikmer**，做**链式扩展**（前/后延伸），要求：链长 ≥ min_shared、位置差 ≤ tolerance、位置差标准差 ≤ max_sd |
| **依赖** | 需要参考基因组预先算好的 unikmer 列表，以及 reads 到 unikmer 的映射（扫描 reads 建索引） |
| **在重复区的表现** | 重复区 unikmer 稀疏，可能因“共享 unikmer 不足”而误删正确 overlap（召回率下降）；你的实验里通过链式扩展 + SD 约束仍能在高重复区保持较好精度 |

### 2. 参考方法：区间哈希（mm2-ivh, Interval Hashing）

| 维度 | 内容 |
|------|------|
| **介入阶段** | **生成 overlap 时**：集成在 minimap2 的 **seed 生成与匹配** 内部，不是后处理 |
| **输入** | 仅 reads（与 minimap2 一致），无需单独 unikmer 文件 |
| **核心思想** | 对**高出现次数的 k-mer**（重复区大量存在）不丢弃，而是把该 k-mer 在序列上**多次出现的间隔（interval）** 编码成哈希。(k-mer, interval_hash) 只有在对齐中“出现模式一致”（2W+1 次出现且间隔一致）才匹配，从而**区分同一 k-mer 在不同重复单元中的出现** |
| **依赖** | 无额外参考；实现上修改了 minimap2 的 sketch/index 与匹配逻辑 |
| **在重复区的表现** | 专门针对重复结构：用间隔模式区分不同重复单元，在 alpha satellite HOR 等场景能显著减少错误 overlap 的**生成量**，从而加快后续步骤 |

## 三、对你的启发

### 1. 定位互补，可明确写进论文

- **区间哈希**：在 **seed 阶段** 减少假阳性 overlap 的**产生**，尤其在高重复区。
- **Ovfilter**：在 **后处理阶段** 用 unikmer 链 + 位置一致性做**精确验证**，与生成 PAF 的工具无关。

在论文“相关工作”或“讨论”中可写：两者解决的是不同阶段的问题，**可组合使用**——例如先用 mm2-ivh 生成 PAF，再用 Ovfilter 做二次校验，在 unikmer 较多的区域多一层保障。

### 2. 实验上的延伸

- 增加一组实验：**Ovfilter 的输入 PAF** 分别来自  
  - minimap2 原始 ava，  
  - mm2-ivh 的 ivh-ava-ont-ul。  
  比较：过滤后保留/丢弃比例、召回率与精确率、总耗时。可说明“区间哈希从源头减假阳 + Ovfilter 再验证”的收益。
- 你第三章表 3-5/3-6 已对比 Raw / Ovfilter / RAfilter / 区间哈希（IH），可在文字中明确点出：**IH 是生成阶段优化，Ovf 是过滤阶段优化**，二者维度不同，组合是自然延伸。

### 3. “间隔”思想的借鉴（可选）

- 区间哈希的本质是：**同一 k-mer 在不同重复单元中，出现间隔不同**，用间隔编码成哈希即可区分。
- 在你当前流程中，可以考虑（仅作思路）：在 unikmer **不足** 的 overlap 上，是否对**非 unikmer 但多次出现的 k-mer** 做“间隔模式”的辅助校验，作为 unikmer 链的补充。实现成本和收益需要单独评估，但可以作为“未来工作”或讨论点。

### 4. 适用场景的区分

- **高重复 + 高精度数据**（如 HiFi、ONT ultra-long）：适合推荐 **mm2-ivh 生成 PAF + Ovfilter 过滤**，既减少错误 overlap 数量，又保留你对 unikmer 的严格验证。
- **一般数据或需兼容现有流程**：Ovfilter 单独使用即可，不依赖 mm2-ivh，与现有 minimap2/其他工具链兼容。

### 5. 局限性写清楚

- **mm2-ivh**：明确写“需要高序列一致性（约 99.5%+）”“对高错误率 reads 效果有限”（见其 README Limitations）。
- **Ovfilter**：明确写“高度重复区 unikmer 稀疏，可能降低召回率”；你的实验已展示随重复比例上升召回率略降，与这一点一致。

## 四、一句话总结

- **区间哈希**：在**生成 overlap** 时，用 k-mer 的**出现间隔**给重复区 k-mer 打“指纹”，从源头减少错误 overlap。
- **你的方法**：在**得到 overlap 之后**，用**唯一 k-mer 的链式扩展与位置一致性**做精确校验。

两者一个在“种子层”、一个在“过滤层”，思路互补；组合使用（mm2-ivh → Ovfilter）可作为你论文中的**方法组合建议**和**后续实验方向**。

## 五、间隔思想在 Ovfilter 中的具体优化方案

下面是在**不改变你现有主流程**的前提下，用“间隔”做**补充校验**的几种可实现方式；目标是在 **unikmer 不足** 时尽量少误杀正确 overlap，同时控制实现和内存/IO 成本。

### 5.1 核心思路回顾

- **区间哈希**：同一 k-mer 在重复单元内多次出现，**相邻两次出现之间的间隔（bp）** 组成一个序列；不同重复单元间隔模式不同，可编码成哈希做“指纹”。
- **在 Ovfilter 里**：当两条 read 在重叠区内**共享同一个非 unikmer 的 k-mer**，且该 k-mer 在重叠区内**两边都出现 ≥ 2W+1 次**时，可比较两边的**间隔序列**是否一致（允许小误差），一致则认为是“同一重复单元”的匹配，可作为 unikmer 链的补充证据。

### 5.2 方案一：按需对重叠区做间隔校验（不扩索引，用序列）

**思路**：仅在 **unikmer 验证失败** 时，对当前 (Q,T) 的**重叠区序列**做一次“间隔救援”：从序列中提取 k-mer 及位置，找“在两边都出现 ≥ 2W+1 次”的 k-mer，算间隔向量并比较，若存在足够多间隔一致的种子再放行。

**步骤**：

1. **前置条件**：过滤时能按 read_id 取到**重叠区序列**（例如从当前已打开的 reads 文件按需读，或维护 read_id → 序列的缓存）。
2. 对当前 PAF 行，若已判为 UnikmerFail（链长 < min_shared 或 SD 超限），则：
   - 取 Q 在 [q_start, q_end]、T 在 [t_start, t_end] 的序列（若 strand 为负，取 T 的反向互补）。
   - 对两段序列分别：滑动窗口取 k-mer，记录 (kmer_hash, 在重叠区内的相对位置)，统计每个 hash 的出现位置列表。
   - 筛选出在 Q 和 T 重叠区内都出现 ≥ 2W+1 次的 k-mer（W 为参数，如 2 或 3）。
   - 对每个这样的 k-mer，在 Q 和 T 中分别按位置排序，计算**相邻位置间隔**得到向量 `dq[]` 和 `dt[]`（长度 2W）。
   - **间隔一致性**：对 `dq` 与 `dt` 做容差比较（例如每对 `|dq[i]-dt[i]| <= tol_bp`，或先粗化再比），一致则计为一个“间隔一致”的种子。
   - 若“间隔一致”的种子数 ≥ 某阈值（如 min_shared_ivh），或能形成一条满足长度/一致性的链，则**改判为通过**（视为“间隔救援通过”）。

**优点**：不扩展现有 unikmer 索引，不加新的大结构；只在 unikmer 失败时多做一次计算。  
**代价**：需要能访问重叠区序列（按需读或缓存），且只对 UnikmerFail 的 pair 做，IO/CPU 增加可控。  
**适用**：重复区 unikmer 稀疏、但重叠区序列可拿到且错误率不高的场景（如 HiFi）。

### 5.3 方案二：扩展索引 —— 为“间隔候选 k-mer”存位置列表

**思路**：在**建 read 索引**时，除 unikmer 外，再对“在参考中出现次数适中”（如 2～24 次）的 k-mer 为每条 read 存储**完整位置列表**；校验时先用 unikmer 链，失败则用这些 k-mer 的间隔模式做二次验证。

**步骤**：

1. **准备“间隔候选”集合**：对参考基因组做 k-mer 计数，将出现次数在 [2, max_rep]（如 max_rep=24）的 k-mer 作为 interval_candidates（可写成一个文件，与 unikmer 文件类似）。
2. **扩展 map_reads_to_unikmers**（或新函数 map_reads_to_kmers）：
   - 仍对 unikmer 只存 (hash, one_position) 的压缩结构（与现有一致）。
   - 对 interval_candidates 中的 k-mer：在同一 read 内若出现多次，存**该 read 内该 hash 的全体位置**（排序后的 array），以便后续在任意 [start,end] 内取子序列算间隔。
3. **校验逻辑**：在 `verify_overlap` 中，若 unikmer 链验证失败，则：
   - 在 [q_start, q_end] 和 [t_start, t_end] 内，从两边的“间隔候选”位置列表中取出子列表；
   - 对每个在两边重叠区内都出现 ≥ 2W+1 次的 k-mer，用位置列表算间隔向量，做容差比较；
   - 若满足“间隔一致”的 k-mer 数或链长 ≥ 阈值，则通过。

**优点**：校验阶段不需要再读序列，全部用已有索引完成，适合对延迟敏感、重复区较多的场景。  
**代价**：索引体积和建索引时间增加（每条 read 多存一批 k-mer 的多个位置）；需要维护 interval_candidates 与 unikmer 两套（或合并为一个“候选 k-mer”列表+类型标记）。

### 5.4 方案三：仅用“间隔一致”做宽松救援（最小实现）

**思路**：不区分 unikmer 与重复 k-mer，在 **UnikmerFail 时**只做一次**轻量级**间隔检查：只要求“存在若干个在重叠区两边都出现多次且间隔模式一致”的 k-mer，不要求它们成链，也不改主索引。

**步骤**：

1. 仅当某条 PAF 被判为 UnikmerFail 时触发。
2. 能拿到 Q、T 在重叠区的序列时（同方案一）：
   - 提取两段序列的 (kmer_hash, position)，找出在两边都出现 ≥ 2W+1 次的 k-mer；
   - 对每个这样的 k-mer 计算间隔向量并做容差比较；
   - 若“间隔一致”的 k-mer 个数 ≥ min_ivh_seeds（如 3～5），则将该 overlap 标为“间隔救援通过”并保留。
3. 可选：在统计/输出里单独记一类 “RescuedByInterval”，便于分析召回提升和假阳率。

**优点**：实现量小，易接到现有 Ovfilter；便于先做消融实验看“间隔救援”对召回/精确率的影响。  
**缺点**：仍依赖重叠区序列的获取；若不做成链、只数种子，理论上假阳略高于“成链+SD”的 unikmer 标准，可通过阈值 min_ivh_seeds 和 tol_bp 调节。

### 5.5 实现时建议的参数与细节

- **W（wing）**：间隔哈希的“半窗长”，即一侧用 W 个间隔，共 2W+1 次出现。W=2 时需 5 次出现，W=3 时需 7 次；重复区容易满足，可先取 W=2。
- **间隔粗化**：mm2-ivh 会对间隔做粗化再哈希，以减少测序误差影响。你可将间隔除以一个 bin 大小（如 50 bp）取整后再比较或哈希，等价于 tol_bp 约 50 bp 级别的容差。
- **只对 UnikmerFail 做**：保证现有“unikmer 通过即通过”的语义不变，间隔只做“救援”，避免在高 unikmer 区域引入多余计算。
- **序列获取**：若 reads 文件很大，建议按 read_id 索引（如 fai 或自建简单索引）后按需 seek 读取重叠区，或对当前批次 PAF 涉及的 read 做小缓存，平衡内存与 IO。

### 5.6 小结与论文可写点

- **方案一**：不改索引，用重叠区序列做“按需间隔救援”，实现和实验最容易上手，适合作为**第一个可做版本**和论文中的“间隔思想初步应用”。
- **方案二**：扩展索引、不读序列即可做间隔验证，适合作为**高性能/生产向**的增强版，可在论文里写为“扩展索引 + 间隔验证”的优化方向。
- **方案三**：最小实现，便于快速验证“间隔救援”是否带来召回率提升，并可与方案一合并（先做方案三，再在方案三基础上加链式或更严条件即接近方案一）。

在论文中可这样表述：在 unikmer 不足的重复区，引入**基于 k-mer 出现间隔的一致性校验**作为补充，与区间哈希在 seed 层的思想一致，但在后处理层以“救援”形式使用，兼顾实现成本与召回率提升；并可在实验中报告“仅 unikmer”“unikmer + 间隔救援”的召回/精确率与运行时间对比。
